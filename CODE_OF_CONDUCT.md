# Code of Conduct

OrcaTrade is a closed-source commercial project. External contributions
are not currently accepted (see [`CONTRIBUTING.md`](CONTRIBUTING.md)).
This file applies to the internal team and to any external party
interacting with this repository's public surfaces (issues, security
reports, pull-request discussions if/when they're enabled).

Last updated: 2026-06-02.

---

## Our standard

We treat each other and every person who interacts with this project as
a colleague — including security researchers, prospective customers,
investors, and anyone who finds the repo through search. The bar is
*professional*, not friendly-or-bust: clear, respectful, focused on the
work, honest about what we know and what we don't.

## Expected behaviour

- Be specific and constructive. Prefer code, data, and citations over
  opinions about people.
- Honour the engineering rules in [`CLAUDE.md`](CLAUDE.md) and
  [`CONTRIBUTING.md`](CONTRIBUTING.md). Disagreements about the rules
  are welcome via PR or written argument; ignoring the rules is not.
- Credit work and ideas. If you build on someone else's commit,
  reference it.
- Disclose conflicts of interest early (financial relationships with
  customers, competitors, or external vendors mentioned in the
  codebase).
- For security findings, follow [`SECURITY.md`](SECURITY.md) — private
  channel first, public disclosure coordinated.

## Unacceptable behaviour

- Harassment, intimidation, or discrimination on any axis (race,
  gender, sexuality, religion, nationality, disability, etc.).
- Personal attacks, name-calling, or ad hominem responses to
  engineering disagreements.
- Public disclosure of a security issue before a fix has shipped
  (regardless of pressure from a third party).
- Sharing customer data, prospect lists, or unreleased product
  information outside the team.
- Deceptive engineering — fabricating test results, falsely claiming a
  rule was followed, misrepresenting the state of a feature in a PR
  description or commit message.

## Reporting

- **Conduct concerns:** email `conduct@orcatrade.pl`. Reports go
  directly to the founder (currently the only senior-most non-conflicted
  contact). Reports are treated as confidential; we will not retaliate
  against good-faith reporters.
- **Security issues:** see [`SECURITY.md`](SECURITY.md) — that channel
  has its own SLA and process.

## Enforcement

The founder is the ultimate decision-maker today. As the team grows,
this will be replaced with a small panel and the process will be
documented here. Current options:

1. **Private feedback** — a one-on-one conversation, the default for
   first-time minor incidents.
2. **Written warning** — for repeated or moderate incidents, with a
   record kept in the team's internal HR notes.
3. **Removal of access** — for severe incidents or repeat patterns,
   the team member loses repo access and (if applicable) employment.
4. **Public ban** — for external bad-faith actors, GitHub-level block
   from the org's repositories.

## Honesty discipline

This file follows the same honesty discipline as
[`docs/security/README.md`](docs/security/README.md): it is dated,
specific about what we have today (a single-founder enforcement path),
and explicit about what we don't have yet (a panel, formal HR
process). We will update it when reality changes, not by adding
marketing-shine but by removing claims that are no longer accurate.
