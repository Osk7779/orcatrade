Section 02

What OrcaTrade is today

OrcaTrade is the operating system for European SMEs importing from Asia. The combination of AI agents, verified supplier infrastructure, and physical presence in Hong Kong moves a buyer from "I don't know how to do this" all the way through to goods on the EU warehouse floor.

2.1 Brand positioning

Find it · Verify it · Ship it · Finance it — one platform, Asia to Europe.
Your import operations team, available 24/7 — that knows every supplier, every shipment, every regulation.

2.2 Ideal Customer Profile (ICP)

Polish, German, and broader Central-European SMEs with annual import volume of 50,000–2,000,000 EUR from China and broader Asia, doing 2–50 shipments per year. Big enough to need help, small enough not to afford an in-house import team and not get the attention of DSV, K+N, or DB Schenker.

Entry segments — first targets:

  • Polish e-commerce founders — particularly Allegro sellers and Amazon FBA operators.
  • German Mittelstand SMEs sourcing manufactured components from Asia.
  • Specialty retailers and brand owners doing 5–50 shipments per year.

2.3 Four elements of competitive advantage (the moat)

  • Proprietary supplier and factory data — accumulated through Sourcing and Factory Search work; competitors cannot replicate this without years of operational effort.
  • Hong Kong physical presence — in-person supplier verification, sample consolidation, dispute resolution. Pure-digital forwarders (Flexport, Forto) cannot match this at the SME price point.
  • EU-specific compliance knowledge — CBAM, EUDR, REACH, CE, RoHS, EU AI Act. Most American AI platforms in this space are US-centric.
  • AI-native architecture from day one — agent infrastructure built into the platform, not bolted on after the fact.

The five-pillar import journey

The entire product is organised around a single strategic frame: the five-stage journey an importer actually walks through. What used to be four scattered products (Factory Search, Sourcing, Intelligence, Finance) has been reframed as stages of one path, with the missing link added: the logistics layer.

3.1 Commercial model

The platform sells through a layered subscription model with transactional revenue on top of execution services:

  • Free — entry point with limits; brand-awareness builder.
  • Starter — 99 EUR / month — SMEs starting to import.
  • Growth — 399 EUR / month — established importers, 5–50 shipments per year.
  • Scale — 999 EUR / month — multi-channel sellers and growing brands.
  • Enterprise — custom pricing — with SLAs and custom integration.

The transactional layer covers insurance commissions (12% baked in), shipment fees in the Freight Forwarding Lite model (8–15% margin), service fees for full import management, and origination fees on buyer verification and sample consolidation work.

Section 04

Platform architecture

4.1 Technology stack

Runtime & hosting: Vercel serverless functions (CommonJS, no npm dependencies in api/); regional edge for static assets; the rate-limit layer lives in lib/intelligence/runtime-store.js.

AI layer: Anthropic API (Claude 4.X family — Opus, Sonnet, Haiku selected by task profile). Strict tool-use loop, streaming SSE, 8-call cap per turn, 30-second timeout per tool invocation, hard human-in-the-loop discipline for irreversible actions.

Frontend: Vanilla JavaScript without a framework — a deliberate choice for simplicity, zero dependencies, and a stable component lifecycle. Cormorant Garant + Geist as the typographic stack. The aurora animated background and cursor-spotlight effects deliver the "wow" layer while remaining print-ready.

Persistence: localStorage for form drafts. Supabase / Auth.js / Clerk are the candidate production authentication providers (the architectural decision remains on the roadmap; the stub-auth skeleton was delivered in Sprint 18).

Other components: Resend for transactional email, embedded BM25 retrieval over the regulatory corpus (pure JS, no external vector-database service).

4.2 The self-contained tool pattern

Every platform tool has been built as a self-contained package (the self-contained tool pattern) replicated eight times to date:

  1. lib/intelligence/[name].js — pure deterministic computation logic with unit tests.
  2. api/[name].js — serverless endpoint with input validation, rate limiting, and a JSON response with a detailed breakdown.
  3. /[name]/index.html — landing page with the product manifesto and a CTA into the tool.
  4. /[name]/quote/index.html — interactive quote form with live recalculation every 200 ms.
  5. test/[name]-quote.test.js — test pack covering constants, validation, scenarios, and recommendations.

The repeatability of this pattern means adding a new tool (Marketplace, Export Services to Asia) is mechanical and predictable. The architecture does not require a "big-bang refactor" for each subsequent roadmap item.

Section 06

Compliance Agent — agent in production

The first of five planned platform agents (Sourcing, Compliance, Logistics, Finance, plus an Operations Orchestrator) has been fully deployed. The agent runs the full Anthropic API tool-use loop with SSE streaming, strict human-in-the-loop discipline for irreversible actions, and a complete audit trail of tool calls.

6.1 The nine agent tools

  1. searchRegulations — BM25 search across the four-regulation corpus.
  2. checkCbamApplicability — does CBAM apply to a given import (HS code, country of origin).
  3. estimateCbamExposure — projected number of CBAM certificates, cost, deadlines.
  4. checkEudrApplicability — commodity coverage and source-country risk.
  5. assessEudrCompliance — Art. 9 due-diligence gaps.
  6. checkReachApplicability — product category and SVHC.
  7. assessReachCompliance — registration path vs. exemption, importer policy.
  8. checkCeApplicability, assessCeCompliance — directive sets, conformity-assessment paths.
  9. lookupHsCode — returns CN8 + description for a natural-language query.
  10. requestHumanReview — escalation to an operator; the only "write" semantic action.

6.2 Runtime discipline

  • Hard cap of 8 tool invocations per turn — bounded loop.
  • 30-second timeout per individual tool call.
  • Rate limiting at 12 requests per minute per IP — cost control.
  • SSE streaming with events: thinking → tool-call → tool-result → text-delta → final → done.
  • Full call-path logging for compliance audit.

6.3 Agent frontend

The /agent/ page provides a full-screen conversational interface with cursor spotlight effect, starter suggestions, tool-call traces with ✓/✗ indicators, and citation chips that link to EUR-Lex. The agent specification (docs/compliance-agent-spec.md) contains the system prompt, nine tool JSON schemas, four sample conversation flows, and requirements covering memory, context, and logging.

Section 08

Tier 2 — Three logistics services

Three of the five Tier 2 items have shipped across sprints 19–21, closing the "physical supply chain" arc — from "how do I ship?" through "what duties apply?" to "where does it sit in the EU?" The remaining two (Marketplace, Export Services to Asia) are still in the queue.

Sprint 19 · Live
Multi-modal Routing — choosing the transport mode

A four-mode comparison engine for Asia–Europe transport (sea FCL / sea LCL / air / rail). Base rates: sea FCL 0.55 EUR/kg, sea LCL 1.40 EUR/kg, rail 1.85 EUR/kg, air 6.50 EUR/kg. Origin multipliers (CN baseline 1.00; IN, VN, BD, ID, TH, MY, PK 1.05–1.30). Rail viability gate: origin ∈ {CN, KZ, KG} AND destination ∈ 17 EU/CEE codes (China–Europe Railway Express via Khorgos / Brest / Małaszewicze).

CO₂ math: 10 / 14 / 600 / 30 g per tonne-km (sea FCL / sea LCL / air / rail). Recommendation engine: urgency <14 days → air; 200–5000 kg China–EU → rail (the sweet spot); ≥5000 kg → sea FCL; <200 kg → air. The railEducation block exposes the Małaszewicze rail corridor — the option most forwarders never pitch despite being 10–15 days faster than sea and 70% cheaper than air.

lib/intelligence/routing-quote.js · api/routing.js · /routing/ + /quote/
Sprint 20 · Live
Customs & Bonded Solutions — clearance and bonded warehousing

A landed-cost estimator with two side-by-side scenarios. Scenario 1: standard clearance — duty (HS chapter MFN) × customs value + import VAT × (customs value + duty) + brokerage (45 EUR base + 8 EUR/line capped 250 EUR) + 25 EUR ENS pre-arrival filing. Scenario 2: bonded warehouse — 95 EUR entry + 0.30 EUR/cbm/day storage + 1.2% bond fee on customs value + 65 EUR exit (or 35 EUR re-export); duty + VAT either deferred (cash-flow benefit at a 6% annual cost-of-capital) or avoided entirely on re-export.

The HS chapter rate table covers ~50 chapters (apparel 12%, footwear 11%, textiles 7–12%, machinery 2.5–3.5%, furniture 2.7%). Origin overlay: Chinese anti-dumping duties (steel chapters 72/73 +18%/+10%, aluminium 76 +10%, footwear 64 +4%) and preferential FTAs (VN EVFTA 70% reduction, BD/KH EBA full exemption, PK GSP+ 50%, TR customs union 95%). Import-VAT rates for all 27 EU member states.

lib/intelligence/customs-quote.js · api/customs.js · /customs/ + /quote/
Sprint 21 · Live
Warehouse / 3PL — six-hub EU benchmark

Benchmark across six EU logistics hubs (Rotterdam NL / Hamburg DE / Frankfurt DE / Poznań PL / Prague CZ / Barcelona ES). Each hub has storage (12–19 EUR/pallet/ month), inbound receipt (9–14 EUR/pallet), pick & pack (per-order base + per-line + per-unit), packaging materials, and a one-off setup fee amortised over 12 months.

Outbound shipping is calculated from a 6×6 region-to-region rate matrix (CENTRAL / NORDIC / IBERIAN / MEDITERRANEAN / EAST / UK) with base + per-kg + transit-day fields. Six value-added services (QC inspection, labelling, kitting, photography, returns processing, gift wrapping). The recommendation engine prefers the cheapest hub when its region matches the primary destination region; otherwise it compares the cheapest against the cheapest-in-destination- region and recommends the faster option only when the premium is ≤10%.

lib/intelligence/warehouse-quote.js · api/warehouse.js · /warehouse/ + /quote/
Section 09

Customer Dashboard — the logged-in product layer skeleton

The first product layer beyond marketing pages and one-shot calculators. The dashboard aggregates state from all seven platform tools (4 document forms + Insurance / Samples / Returns), groups activity by resource type (documents, quotes), and surfaces the relevant stats and action paths.

9.1 Components delivered in Sprint 18

  • lib/dashboard/state-aggregator.js — an environment-agnostic state aggregator (Node-testable, browser-runnable). Reads localStorage drafts, summarises each (invoice number, parties, line count, currency for documents; cargo, mode, route for quotes). Session helpers buildSession, saveSession, loadSession, clearSession with role validation.
  • dashboard/index.html — sidebar layout with four navigation sections (Workspace / Documents / Tier 1 services / Pillars), a topbar showing workspace name + user + tier, a stats row (drafts / documents / quotes / plan tier), document and quote panels, a 6-up quick-actions grid, and a roadmap panel.
  • Stub auth — dashboard/auth.js + /login/ + /signup/. Browser-side, with explicit demo-mode banners. The OrcaAuth API mirrors what will be replaced by a production integration (Supabase / Auth.js / Clerk).

9.2 The next architectural decision

Choosing the authentication provider and migrating from localStorage to server-persisted state is a deliberately deferred decision. The strategic plan calls this out as the "next architectural decision" — worth being made consciously after gathering signals from initial pilots, rather than committed to in isolation.

The test pack uses Node's built-in node:test with no external frameworks, preserving the zero-dependency policy of the lib/intelligence/* modules. Tests cover input validation, recommendation scenarios, pricing-snapshot constants, mathematical identities (e.g. landed cost = customs value + duty + VAT), and edge cases.

10.1 Agent eval harness

The file test/agent-eval-cases.json (v1.1) contains 15 scenarios: 3× CBAM, 3× EUDR, 2× REACH, 3× CE, 1× multi-regulation, 3× escalation paths, 2× out-of-scope. The script scripts/agent-eval.js runs cases against the agent handler, asserts called tools, citation presence, keywords, escalation, and stop reason. --bail and single-case-by-id are supported; exit codes are CI-suitable.

10.2 Engineering discipline

  • Zero-dependency policy in the computational layer — pure JS, no npm packages in production api/ modules.
  • Deterministic calculations — every form shows the formula and the math breakdown to the customer.
  • Pricing snapshot in every module with asOf and source fields — auditable and refreshed quarterly.
  • Validation at system boundaries — internal code trusts data, validation lives at API and UI inputs.
  • Per-IP, per-endpoint rate limiting — cost control on Anthropic API.

12.1 Strategic conclusions

After eleven sprints, the platform is at a stage where its product base covers the entire material span of the five-stage import journey. Each of the nine tools functions independently while simultaneously closing the larger whole. Engineering discipline (zero-dep, deterministic calculations, pricing snapshots, 381 tests) ensures that adding subsequent tools does not generate technical debt.

Three signals indicating project maturity:

  1. Repeatability of the implementation pattern — each of the three Tier 2 tools shipped in a single sprint with an identical package (lib + api + landing + quote + tests). The unit-cost of adding a new tool is predictable.
  2. Architecture holds up under growth — the navigation refactor (Sprint 22) was needed but cheap. A good signal that the architecture is not falling into a technical-debt trap as features accumulate.
  3. No gap in the compliance layer — where competitors focus on a single regulation (most often CBAM), OrcaTrade has four in production with a conversational agent able to orchestrate them in a single client conversation.
The direction remains unchanged — the project's north star, set by the founder, is "let's make orcatrade big." The platform thesis finds confirmation in shipped product. The next move is the disciplined delivery of the remaining roadmap items and the first commercial pilots with Polish and German SMEs in the e-commerce segment.
